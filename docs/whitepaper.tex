\begin{abstract}
This document is a compact tutorial and research note about co-designing a neural
network for the AMD XDNA~2 NPU. The working example is deliberately simple: a
32-layer residual MLP, first mapped to MNIST and then retargeted to CIFAR-10,
with one learnable weight matrix per compute tile and activations flowing
through the full 32-tile array in a snake. The point is not that these tiny
vision tasks are intrinsically hard, nor that the current design reaches the
hardware's peak throughput --- it does not. The point is that it is small
enough to understand end to end, yet rich enough to expose the real constraints
that matter on this hardware: local SRAM, MMUL-friendly data layouts, explicit
FIFOs, and the cost of host--device orchestration.
\end{abstract}

\section{Introduction}

Modern NPUs are most interesting when they stop looking like ``small GPUs'' and
start looking like what they are: spatial dataflow machines. On AMD XDNA~2, the
programmer sees a 2D array of compute tiles, each with local SRAM, a VLIW + SIMD
core, a matrix-multiply unit, and explicit data-movement primitives. That makes
this hardware unusually transparent. It also means that a model that looks
natural in PyTorch may look awkward once you try to map it directly onto the
array.

This repository takes the hardware seriously. Instead of beginning with a large
model and then asking how to squeeze it onto the device, we begin with the tile
array and ask: what is the cleanest learnable network that maps onto it almost
literally? The answer in the current main branch is a 32-layer residual MLP
with one residual weight matrix per tile:
\[
  h_{\ell+1} = \mathrm{ReLU}(h_{\ell} W_{\ell}) + h_{\ell}, \qquad \ell=0,\dots,31.
\]
Each of the 32 compute tiles stores exactly one residual weight matrix. In the
main public MNIST $H=160$ split, the host CPU still runs the input embedding,
while the NPU runs all 32 hidden layers and the final classification head in
one streamed service. The same residual template can then be retargeted to
CIFAR-10 by widening the host-side input projection and moving to a narrower,
higher-batch throughput point.

The result is now a family of didactic but fully working examples:
\begin{itemize}
  \item the main $H=160$ end-to-end path reaches about 488{,}597 images/s,
        or roughly $3.5\times$ over pure CPU inference on the same host,
  \item the resident $H=160$, $B=8$ residual body reaches about 1.46 million
        images/s wall-clock at \code{stream\_depth=32}, or about 2.39 TFLOPS,
  \item a narrower throughput-oriented $H=96$, $B=56$ variant reaches about
        97.3\% validation accuracy on a 50k/10k train/validation split, about
        739k images/s wall-clock in the current host-visible service, and about
        3.88 million images/s if only NPU kernel wait time is counted,
  \item a throughput-tuned CIFAR-10 $H=32$, $B=176$, \code{stream\_depth=64}
        variant sustains about 747{,}851 images/s wall-clock over a 100M-image
        run and about 3.06 million images/s if only NPU kernel wait time is
        counted, with the CPU-side $3072\rightarrow32$ embed now dominating the
        total wall time.
\end{itemize}

These numbers are intentionally modest relative to the NPU's peak. The current
paper is therefore not a claim that this is the fastest possible use of XDNA~2.
It is a clear case study in co-design: how the shape of a network changes once
local memory, tiling, and routing become first-class design constraints. Older,
more experimental iterations of the project are preserved in \code{logbook.md}
and on the \code{experimental} branch; this whitepaper focuses only on the
current \code{resmlp/} implementation.

\section{The hardware model we program against}

\subsection{Why spatial dataflow changes the design problem}

A CPU or GPU programmer usually thinks in terms of a few large kernels operating
against a shared memory system. XDNA~2 rewards a different mental model: keep
small state local to each tile, move only what must move, and make the routing
pattern of the computation explicit.

\begin{table}[t]
\centering
\small
\begin{tabularx}{\textwidth}{@{}>{\bfseries}p{0.22\textwidth}XX@{}}
\toprule
Mental model & CPU / GPU intuition & XDNA~2 intuition \\
\midrule
Compute & A few large kernels over shared memory & Many small tile-local kernels connected by explicit streams \\
State & Parameters and activations live in DRAM / cache / HBM & Parameters should fit in tile SRAM whenever possible \\
Data movement & Often implicit (cache hierarchy, kernel launch runtime) & Explicit (DMA, ObjectFIFOs, placement, routing) \\
Architecture choice & Mostly driven by model quality & Driven by both model quality and physical layout \\
\bottomrule
\end{tabularx}
\caption{The whitepaper assumes the reader is comfortable with neural networks, but new to spatial dataflow hardware. The central shift is from ``launch bigger kernels'' to ``shape the network so each tile has a clean local job.''}
\label{tab:cpu-vs-npu}
\end{table}

For this reason, the first design question in this project is not ``what is the
best architecture for MNIST?'' but rather ``what architecture maps cleanly onto a
32-tile array with \textasciitilde{}64 KB of SRAM per tile?'' That shift is
what makes the project interesting.

Three remarks are worth keeping in mind.
\begin{enumerate}
  \item The most valuable resource is not peak arithmetic throughput; it is the
        ability to keep parameters local in SRAM.
  \item The MMUL datapath likes blocked 8$\times$8 layouts, which is why this
        project insists on dimensions divisible by 8 and uses explicit tiling on
        the host side.
  \item Data movement is explicit. Activations do not ``just appear'' on the next
        tile. They move because FIFOs and DMA descriptors were created for them.
\end{enumerate}

\section{A network that matches the array}

\subsection{Model overview}

The current model is intentionally plain. In the main $H=160$ streaming path,
the CPU flattens a 28$\times$28 MNIST image and projects it to a
160-dimensional hidden state. On the NPU, we apply 32 copies of the same layer
shape, each with its own weights, and the final tile also projects the last
hidden state to ten logits. Later subsections revisit the same 32-layer
template at narrower, higher-batch operating points for both MNIST
($H=96$, $B=56$) and CIFAR-10 ($3072\rightarrow32$, $B=176$,
\code{stream\_depth=64}).

For the main $H=160$ streaming path, this split is not arbitrary. The embedding
stays on the CPU because a one-tile resident $784\rightarrow160$ projection is
not SRAM-feasible, while the hidden stack and the small $160\rightarrow10$ head
fit comfortably on-array. The hidden stack is the repetitive part: it is where
a tile array is easiest to use well, because the same buffer shape recurs 32
times.

The PyTorch model in \code{resmlp/model.py} is therefore almost identical to the
hardware view:
\begin{lstlisting}[language=Python]
class ResidualLinear(nn.Module):
    def forward(self, x):
        return torch.relu(x @ self.weight) + x

class ResMLP(nn.Module):
    def forward(self, x):
        x = x.view(x.size(0), -1)
        x = self.embed(x)
        for layer in self.layers:
            x = layer(x)
        return self.head(x)
\end{lstlisting}

That symmetry matters. The main reason this whitepaper is easy to write at all is
that the conceptual model and the deployed model are so close to one another.

branch. Instead, they are chained into one long serpentine path. The activation
enters the first tile, flows through all 32 layers, and leaves the last tile.
This is the cleanest way to make ``one weight matrix per tile'' literally true.

In code, this ordering is generated by a tiny helper:
\begin{lstlisting}[language=Python]
def snake_tile_order(num_cols):
    tiles = []
    for col in range(num_cols):
        rows = range(2, 6) if col % 2 == 0 else range(5, 1, -1)
        for row in rows:
            tiles.append((col, row))
    return tiles
\end{lstlisting}

That function is deceptively simple. It is the point where the network
architecture stops being abstract and starts matching the physical device.

\subsection{Why \texorpdfstring{$H=160$}{H=160} and \texorpdfstring{$B=8$}{B=8}}

The hidden dimension $H=160$ is chosen by SRAM, not by theory. A single tile must
hold one $160\times160$ bf16 weight matrix plus the input and output activation
buffers for a batch of 8 samples. Figure~\ref{fig:sram-budget} shows the
budget.

Numerically:
\begin{itemize}
  \item weight matrix: $160\times160\times2 = 51{,}200$ bytes $\approx 50.0$ KB,
  \item input activations: $8\times160\times2 = 2{,}560$ bytes,
  \item output activations: $8\times160\times2 = 2{,}560$ bytes.
\end{itemize}
This lands at roughly 55 KB before stack and miscellaneous overhead, leaving
comfortable headroom under the tile's local memory limit.

The batch size $B=8$ follows from the MMUL granularity. The current kernel uses
8$\times$8$\times$8 blocking and expects dimensions divisible by 8. Choosing
$B=8$ is therefore the smallest clean batch that does not waste the MMUL shape.
It is emphatically not the throughput-maximizing choice. It is the smallest
choice that keeps the mapping simple and the tutorial honest.

\subsection{BFP16 and tiled host layouts}

The main branch compiles the kernel with
\code{-DAIE\_API\_EMULATE\_BFLOAT16\_MMUL\_WITH\_BFP16}. In practice this means
that host-side weights and activations are arranged into blocked 8$\times$8
layouts, and the MMUL path uses a BFP16-style execution mode that aligns well
with that blocking. The helper functions \code{to\_tiled()} and
\code{from\_tiled()} in \code{resmlp/\_\_init\_\_.py} perform the required data
reordering.

This choice trades a small amount of numerical fidelity for a much cleaner and
more uniform hardware mapping. In the test harness, the difference between the
NPU path and a float32 reference compounds across 32 layers into noticeable
per-element mismatch, but it does not meaningfully degrade classification
accuracy.

\section{From network to NPU program}

\subsection{What one tile computes}

The tile-local kernel in \code{aie\_kernels/matmul\_relu\_skip.cc} implements the
whole layer in two steps: a blocked matrix multiply with ReLU, followed by a
vectorized residual add.

\begin{lstlisting}[language=C++]
extern "C" void matmul_relu_skip_bf16(bfloat16 *a, bfloat16 *w, bfloat16 *c)
{
    static_assert(DIM_M % 8 == 0, "Batch must be multiple of 8");
    static_assert(DIM_K % 8 == 0, "Input dim must be multiple of 8");
    static_assert(DIM_N % 8 == 0, "Output dim must be multiple of 8");

    ::aie::set_rounding(aie::rounding_mode::floor);
    matmul_relu<bfloat16, (DIM_M / 8), (DIM_K / 8), (DIM_N / 8)>(a, w, c);
    residual_add(a, c);
}
\end{lstlisting}

Two details matter here.
\begin{enumerate}
  \item The current kernel is deliberately simple. It uses a 1$\times$1 output
        blocking pattern rather than the more aggressive 2$\times$2 expansion used
        in older throughput-oriented experiments.
  \item The residual connection is explicit in the kernel. This keeps the NPU
        path numerically close to the PyTorch model instead of forcing the host
        to perform an extra add between layers.
\end{enumerate}

\subsection{How IRON describes the snake}

The IRON design in \code{resmlp/design.py} mirrors the conceptual picture almost
literally:
\begin{itemize}
  \item one activation FIFO into the first tile,
  \item one activation FIFO out of the last tile,
  \item one activation FIFO between every pair of consecutive snake stages,
  \item one weight FIFO per column, split in the MemTile into four tile-local
        weight streams,
  \item one worker per compute tile.
\end{itemize}

The worker body itself is tiny:
\begin{lstlisting}[language=Python]
def worker_fn(of_in, of_out, of_w, kern):
    x = of_in.acquire(1)
    y = of_out.acquire(1)
    w = of_w.acquire(1)
    kern(x, w, y)
    of_w.release(1)
    of_in.release(1)
    of_out.release(1)
\end{lstlisting}

This is one of the nicest properties of the current implementation. There is no
large orchestration layer hidden elsewhere. The dataflow graph is already the
program.

An important subtlety is that the code uses plain \code{ObjectFifo}s for the
activation chain, even across column boundaries. The compiler handles the
required inter-column routing through the memory infrastructure; the Python code
does not manually spell out every routing hop.

\subsection{What moves during one inference call}

The end-to-end inference path in \code{resmlp/infer.py} is:
\begin{enumerate}
  \item load a trained PyTorch checkpoint,
  \item run the CPU embedding layer,
  \item convert the hidden activations to tiled bf16 layout,
  \item write input activations and packed weights to the NPU buffers,
  \item execute the NPU runlist,
  \item read back the final hidden state and apply the CPU classification head.
\end{enumerate}

The accuracy result is the easy part: the hardware path matches the software

The speed result needs more interpretation. A batch of 8 through 32
hidden layers performs roughly
\[
  32 \times \bigl(2 B H^2 + B H\bigr)
  \approx 13.15\text{ million operations}
\]
for $B=8$ and $H=160$, counting both the matrix multiply and the residual add.
That is not much work for a device with a 25-TFLOP bf16 headline. The design is
therefore dominated by overheads: invocation cost, buffer writes, and the simple
fact that $B=8$ was chosen for clarity, not for saturation.

\subsection{Why the current design is far from peak throughput}

The right way to read the current performance is not ``the NPU is slow.'' The
right way to read it is ``this particular example is optimized for transparency.''
Three factors dominate.
\begin{enumerate}
  \item \textbf{Small batch.} $B=8$ is the smallest neat MMUL-aligned batch, but it
        leaves little work to amortize fixed costs.
  \item \textbf{Per-batch weight writes.} The current runtime writes the entire
        1.56 MB weight buffer every batch, even though the weights do not change.
  \item \textbf{Simple kernel scheduling.} The main branch uses the simplest clean
        kernel and routing strategy, not the most aggressively throughput-tuned one.
\end{enumerate}

This is why the paper should not be read as an argument that ``the best use of
XDNA~2 is MNIST with $B=8$.'' It is instead an argument that a clean end-to-end
example can make the co-design problem legible.

\subsection{Why the example is still valuable}

Despite the modest utilization, the current design teaches several things very
cleanly.
\begin{itemize}
  \item A neural network can map directly to the physical tile graph with very
        little abstraction gap.
  \item The most important design question on this hardware is often SRAM budget,
        not parameter count in the abstract.
  \item A simple CPU/NPU split --- CPU for irregular edges, NPU for the repetitive
        core --- is often the cleanest way to proceed.
  \item Explicit data movement clarifies the true cost model: what stays local,
        what streams, and what the host still has to do.
\end{itemize}

That is precisely why this repository pivoted away from more fragmented earlier
iterations and toward the current \code{resmlp/} example.

\section{Conclusion and next directions}

The cleanest story in this repository is not ``we reached peak TOPS.'' It is
``we built a neural network whose structure is dictated by the hardware, and we
can explain every part of that mapping.'' That is already a useful result.

For future work, the next steps are straightforward:
\begin{itemize}
  \item keep weights resident across calls instead of rewriting them every batch,
  \item increase batch size to better amortize invocation overhead,
  \item explore int8 variants,
  \item move to a task that is more representative than MNIST once the mapping is
        fully understood.
\end{itemize}

The older experimental history remains valuable --- especially the throughput
benchmarks and language-model work --- but it belongs in \code{logbook.md} and the
\code{experimental} branch. The main whitepaper should stay sharp: one device,
one mapping, one working example, fully explained.

\section{Current full-NPU training implementation}

The main text explains the cleaner mapping that the current whitepaper is built
around: CPU embedding, a 32-tile residual snake on the NPU, then a CPU
classification head. The code in this repository now also contains a more
ambitious training path in \code{resmlp/train\_npu.py}. That path moves the
entire MNIST model onto the NPU for one training call: embedding, residual
stack, classification head, loss, backward pass, and SGD updates.

This appendix documents that implementation as it exists today. The important
point is not that every systems problem is solved. It is that the training
computation itself now lives on the array. The remaining rough edges are mostly
runtime-state and persistence problems, not missing backpropagation math.

\subsection{Tile layout and layer mapping}

The full training graph still uses all 32 compute tiles, but not in the same
role assignment as the inference snake from the main text. In
\code{resmlp/training\_full\_design.py}, the 32 tiles are partitioned as shown in
Table~\ref{tab:full-training-tiles}.

This means that \code{full-npu} is a 30-residual-layer training design, not the
older 32-residual-layer inference snake described earlier. The difference is
structural, not cosmetic: the two end tiles are now used for the irregular
edges of the network instead of being forced into the residual pattern.

\subsection{What one training call does on the array}

One invocation of the full training program performs the whole forward and
backward path on device.
\begin{enumerate}
  \item The host streams a raw $8\times784$ MNIST batch into the embed tile.
  \item The embed tile applies the $784\rightarrow32$ projection.
  \item The activation flows through 30 residual tiles in snake order.
  \item The head tile computes logits, softmax-cross-entropy quantities, class
        predictions, and the tiled \code{d\_logits} buffer.
  \item The backward pass starts at the head tile and propagates gradients back
        through the residual chain to the embed tile.
  \item Each stage applies its own SGD update in place.
\end{enumerate}

The backward kernels are the main novelty. The head kernel in
\code{aie\_kernels/head\_backward.cc} computes
\[
  dY_{\text{hidden}} = d\text{logits}\, W_{\text{head}}^{T}
\]
and then applies
\[
  W_{\text{head}} \leftarrow W_{\text{head}} - \eta\, Y_{\text{hidden}}^{T} d\text{logits}.
\]
The residual kernel in \code{aie\_kernels/residual\_backward.cc} performs both
gradient propagation through the skip-ReLU layer and the outer-product SGD
update of the local $32\times32$ weight matrix. The embed kernel in
\code{aie\_kernels/embed\_backward.cc} does the same for the $784\times32$
projection matrix.

In other words, the appendix-level claim is strong but precise: SGD is not
simulated on the host after an NPU forward pass. The weight updates themselves
are executed inside the AIE kernels.

\subsection{How IRON wires the training graph}

The IRON graph in \code{resmlp/training\_full\_design.py} is the clearest place
to see how the training step is organized.
\begin{itemize}
  \item \code{embed\_wt} and \code{head\_wt} are direct weight FIFOs.
  \item Residual weights enter as eight per-column FIFOs and are split in the
        MemTiles into 3 or 4 tile-local weight streams, matching the snake
        occupancy of each column.
  \item Activations form one forward chain of \code{ObjectFifo}s.
  \item Gradients form one backward chain of \code{ObjectFifo}s.
  \item Each residual tile stores a compact checkpoint buffer containing its
        input activation and ReLU mask, so the backward phase can reconstruct the
        local gradient update.
  \item A small \code{done\_fifo} marks backward completion, and
        \code{head\_wt\_out} drains the updated head weights back to host-visible
        memory.
\end{itemize}

One subtle but important detail is the embed backward phase. The embed tile does
not keep the full raw $8\times784$ batch as a forward checkpoint in local SRAM.
Instead, the host re-streams \code{x\_raw} during the backward phase. This is a
good example of the actual programming style on the array: whenever a buffer is
too awkward to keep resident, the design should make that data movement
explicit.

\subsection{What works today, and what does not}

The current implementation is good enough to support the central claim of this
appendix: the full network, the backward pass, and the SGD update rules can all
run on the NPU in one training call. It is not yet a fully resident training
loop across many batches.

The stable path in \code{run\_full\_npu\_batch()} still uses a fresh
\code{AIEContext} for every batch. That workaround remains necessary because
reusing the same hardware context corrupts later invocations. The important
refinement is that the wrapper no longer needs a full
\code{DefaultNPURuntime.cleanup()} on every call. Evicting only the cached
hardware contexts is enough to restore correct execution while keeping the
instruction-buffer cache hot. Long runs still need a small amount of host-side
garbage collection to release the invalidated contexts. In the current code,
collecting every 256 full-NPU batches is enough to stay stable well past the
two-thousand-context mark where a pure no-GC wrapper starts to fail.

In the current stable configuration, host-visible persistence works reliably for
\code{head\_wt} only. The embed and residual weights are updated in tile memory
during a call, but the system does not yet drain them back in a form that can be
reused safely across calls. Earlier attempts either exceeded SRAM, failed
routing, or exhausted DMA descriptor resources.

This changes the performance picture substantially. The NPU work itself still
takes about 5 ms for the $B=8$, $H=32$ training design, but the wrapper around
it is now only a small extra cost instead of a 250 ms teardown penalty. A
limited 500-batch probe now completes in about 3.7 s while preserving the same
learning behavior as the older fresh-runtime wrapper. So the present result
should be read as both a correctness milestone and evidence that the runtime
overhead is no longer the dominant bottleneck for this path.

Even with that limitation, the path is no longer a dummy stub. In the current
repository, the stable full-NPU training mode learns on MNIST instead of
remaining at chance. After adding the periodic-GC safeguard, a full two-epoch
MNIST run completes on hardware. In one representative run, epoch 0 takes
about 77.6 s, epoch 1 about 125.3 s, and the final host-side accuracies are
about 44.9\% on the training set and 45.6\% on the test set. The remaining gap
to a cleaner throughput story comes from host-side fresh-context orchestration
and missing full weight persistence, not from missing forward, backward, or SGD
math on the array.

\section{One-column simple convnet bring-up}

To separate ``training math works on the array'' from the harder residual-state
and persistence problems, the repository now also contains a deliberately small
alternative path in \code{simplecnn/}. This design uses only one compute column
and four trainable tiles:
\begin{itemize}
  \item tile 0: a $1\rightarrow4$ convolution, $3\times3$, stride 2, producing
        $14\times14\times4$,
  \item tile 1: a $4\rightarrow8$ convolution, $3\times3$, stride 2, producing
        $7\times7\times8$,
  \item tile 2: an $8\rightarrow16$ convolution, $3\times3$, stride 2,
        producing $4\times4\times16$, followed by global average pooling,
  \item tile 3: a linear $16\rightarrow10$ classifier with a tiny top-1 hinge
        update.
\end{itemize}

This path started as a compile-oriented simplification, but it still needed a
surprising amount of correctness work before the metrics meant anything. The
generic conv training kernel overflowed tile program memory, so the final design
uses three fixed-shape kernels (\code{conv1\_train.cc},
\code{conv2\_train.cc}, \code{conv3\_train.cc}). More importantly, the model
and the kernels had to agree on several non-obvious conventions:
\begin{itemize}
  \item conv weights are packed as \code{[out, ky, kx, in]}, not PyTorch's
        native \code{[out, in, ky, kx]},
  \item the pooled $8\times16$ buffer must stay row-major all the way into the
        classifier head, not be silently reinterpreted as an $8\times8$ tiled
        matrix,
  \item generated MLIR / XCLBIN / instruction artifacts must fingerprint kernel
        source changes as well as the Python design files, otherwise stale
        binaries can survive apparently real rebuilds,
  \item conv weight drainback needs explicit \code{wait=True}; without it, the
        head weights drained correctly while the conv weights arrived on the host
        as zeros.
\end{itemize}

After these fixes, one-batch NPU updates matched a bf16-quantized CPU replay of
the same kernels closely, and the batch predictions matched exactly. That made
the remaining limitation much clearer: the one-column convnet is now a
parity-checked, stable full-NPU training path, but it is not yet a good MNIST
model.

\subsection{Resident-window follow-up}

After the 10-epoch bring-up, the next question was whether the one-column path
could follow the same broad strategy used in AMD's official one-column streaming
examples. The closer analogue is the edge-detect example: one task pipeline over
a single compute column, using neighboring-tile memories and ping-pong-style
buffering to reduce dead time between kernels. The scaled color-threshold
example is a useful secondary reference, but it is primarily a MemTile
split/concat design for data-parallel fanout and fanin, not a layer-parallel
training pipeline.

The practical adaptation for \code{simplecnn} was therefore at the
host/runtime boundary first. The current CLI now groups minibatches into
resident windows, keeps one shared host/device \code{weights} BO live for the
whole window, and uses per-batch taps to address slices of larger host buffers.
Crucially, the tile-side \code{ObjectFifo} payloads remain one-batch wide:
making the tile FIFOs themselves window-sized immediately overflowed local
SRAM, so the only scalable pattern is ``large host buffers, small tile buffers,
repeated taps''. The current conservative default is
\code{--window-batches 5}.

\section{Deep streaming residual inference}

The most encouraging recent result is that the repository now has a genuinely
useful inference-only streaming path for the original 32-tile residual snake.
The key simplification is to stop treating inference like training. For
training, weights and ReLU state need to move in and out of the array and the
host has to preserve backward-facing context. For inference, none of that is
necessary. Each tile can simply hold its own fixed checkpoint weight matrix in
SRAM, consume one activation microbatch, emit one activation microbatch, and do
so repeatedly.

This gives a design that is still easy to describe:
\begin{itemize}
  \item the CPU keeps the image flattening, embedding, and classifier head,
  \item the NPU keeps the 32-layer residual body,
  \item each compute tile stores one blocked $160\times160$ bf16 weight matrix
        as a compile-time initialized local \code{Buffer},
  \item the host-facing runtime contract shrinks to just \code{input} and
        \code{output} buffers,
  \item repeated calls reuse the same instructions and the same embedded
        weights, so the host no longer re-sends the residual body parameters
        every batch.
\end{itemize}

\subsection{What changed in the implementation}

Two details mattered much more than they first appeared.

First, inference needed its own kernel entrypoint. The original
\code{matmul\_relu\_skip.cc} kernel had been extended for training so that it
could also write a ReLU mask for the backward pass. That is correct for the
training graph, but inference only wants the three logical tensors
\code{(a, w, c)}. Reusing the five-argument training entrypoint in an
inference graph silently scribbled mask data into unrelated buffers and made
deep streaming look unstable long before the dataflow itself was the real
problem. The final solution is a dedicated three-argument
\code{matmul\_relu\_skip\_infer\_bf16} wrapper for inference while preserving
the mask-writing entrypoint for training.

Second, artifact naming had to become more disciplined. Once the same source
file can produce both training and inference entrypoints across several hidden
sizes, short names like ``\code{resmlp\_kernel.a}'' or
``\code{resmlp\_streaming\_kernel.a}'' are no longer safe. The current
operators fingerprint the relevant Python design files, kernel source, and, for
streaming inference, the packed checkpoint weights themselves. That avoids the
particularly confusing failure mode where an apparently fresh rebuild links a
stale object compiled for a different hidden size or ABI.

\subsection{From resident weights to deep streaming}

With weights embedded, the first gain is easy to understand: even at
\code{stream\_depth=1}, the design is already much faster than the old one-shot
path because the host no longer pushes all 32 layer weights through DDR and the
MemTiles on every call. The next step is to let one call carry several
microbatches. In the current implementation, \code{stream\_depth} means exactly
that: how many $B=8$ activation microbatches are fed through the snake before
control returns to the host.

The first implementation of this idea still hit an avoidable runtime ceiling.
Even though the array-side workers were happy to loop over many microbatches,
the host runtime issued one fill task and one drain task per microbatch. That
kept the boundary FIFOs small, but it meant that deeper chunks stressed the
shim-task schedule before they stressed the arithmetic.

The fix was to keep the tile-local FIFOs exactly as small as before
(\code{depth=2} for \code{stream\_depth > 1}) and instead change the boundary
runtime contract. The current design uses a single repeated shim DMA fill and a
single repeated shim DMA drain per call, driven by a custom
\code{TensorAccessPattern} whose leading size is \code{stream\_depth}. In
practice, that means one runtime task can replay the same $B\times H$
microbatch transfer many times, so the first and last compute tiles never have
to allocate a giant activation window in SRAM just to reach a deeper steady
state.

This is still not a fully free-running conveyor belt. Control still returns to
the host after a finite chunk, and a true indefinitely running service would
need a more persistent runtime abstraction. But inside that chunk, the array now
does behave like the desired layered pipeline: while later tiles are working on
older microbatches, earlier tiles can already start the next ones. The measured
throughput scaling confirms that this overlap is real all the way to
\code{stream\_depth=32}.

\subsection{What depth 32 means --- and what it does not}

At steady state, \code{stream\_depth=32} finally realizes the intuitive
conveyor-belt picture for this 32-layer snake: each compute tile can be working
on a different $B=8$ microbatch at the same time. That is the occupancy target
the earlier finite-task runtime could not safely reach.

What this does \emph{not} mean is that the whole MNIST inference graph is now
on the NPU. The current public $H=160$ streaming path still keeps image
flattening and embedding on the CPU; the residual body and classifier head now
run on the NPU. That is a better end-to-end boundary than the earlier CPU-head
split, but it still does not eliminate host work entirely: the residual-body
measurements above are best read as the throughput ceiling of the resident
snake itself, not as the wall-clock throughput of the full host-visible
inference service.

The conclusion is encouraging, but also clarifying. For the original
$H=160$ design, the NPU path now delivers a real end-to-end gain of roughly
$3.5\times$ over pure CPU inference on this host. Moving the head onto the last
tile also buys a modest but measurable gain over the earlier CPU-head split.
The resident 32-tile snake is
therefore not just a pretty internal benchmark; it does translate into a
visible wall-clock speedup even after host-side boundary work is included.

The $H=32$ prototype tells a different story. Moving the first layer onto tile
0 is architecturally clean and it does work reliably at
\code{stream\_depth=32}, but it is not yet a throughput win. The first tile now
spends real time on the $784 \rightarrow 32$ projection, and the service still
has to untile the final hidden state and run the classifier head on the CPU. In
other words, endpoint migration is not automatically synonymous with higher
throughput; it only helps if the new on-array stage is cheaper than the host
work it replaces.

\subsection{A throughput-oriented \texorpdfstring{$H=96$, $B=56$}{H=96, B=56} point}

The same 32-layer residual template can also be pushed toward a more
throughput-friendly point on the SRAM frontier by narrowing the hidden state and
growing the microbatch. The best current example uses a
$784 \rightarrow 96$ embedding, 32 residual layers of shape $96\times96$, and a
$96 \rightarrow 10$ head. That model has $784\cdot96 + 96 = 75{,}360$
parameters in the embed, $32\cdot96\cdot96 = 294{,}912$ in the residual body,
and $96\cdot10 + 10 = 970$ in the head, for a total of 371{,}242 trainable
parameters. One forward pass costs about 742{,}272 FLOPs/image if only the
matrix multiplies are counted, or about 748{,}522 scalar floating-point
operations/image if bias adds, ReLUs, and residual adds are included.

On a 50k/10k MNIST train/validation split, a simple CPU-trained checkpoint
reaches 97.27\% validation accuracy, and the streamed inference service
reproduces 97.25\% through the NPU residual body. Using $B=56$ and
\code{stream\_depth=32}, the repeated-image service benchmark processes about
739{,}473 images/s wall-clock in the current host-visible path and about 3.88
million images/s if only NPU kernel wait time is counted, corresponding to
about 0.55 / 2.91 TFLOPS if the whole network forward pass is charged to the
measurement. For this shape the public fused logits tail is not yet stable, so
the present benchmark uses the correct body-only streamer with CPU embedding and
CPU head in the Python service loop.

We do \emph{not} claim this is a universal literature record. Direct MNIST
throughput comparisons are surprisingly scarce and often differ in model family,
quantization, batching, or whether host-side work is included at all. Public
benchmark collections such as QuTiBench and GPU/TPU batch-size studies report
MNIST inference regimes in the hundreds of thousands of images/s on very
different systems \cite{qutibench-mnist,batch-size-gpu-tpu}. Our measured
wall-clock figure therefore appears to be at least at the high end of the
public MNIST throughput numbers we could find, but it should be read as a
strong data point rather than an apples-to-apples claim of beating every
published result.

\subsection{A sustained CIFAR-10 throughput point}

The same residual template also retargets cleanly to CIFAR-10. The highest
wall-clock point we measured keeps the 32-tile snake intact, but changes the
host-side projection to $3072\rightarrow32$, uses 32 residual layers of shape
$32\times32$, and keeps the small $32\rightarrow10$ head on the final tile.
That model has $3072\cdot32 + 32 = 98{,}336$ parameters in the embed,
$32\cdot32\cdot32 = 32{,}768$ in the residual body, and $32\cdot10 + 10 = 330$
in the head, for a total of 131{,}434 trainable parameters. One forward pass
costs 262{,}784 FLOPs/image if only the matrix multiplies are counted, or about
264{,}874 scalar floating-point operations/image if bias adds, ReLUs, and
residual adds are included.

Training this smaller model on CIFAR-10 is predictably less accurate than the
wider MNIST-oriented shapes, but it remains nontrivial: the best checkpoint in
the current run reaches 51.43\% validation accuracy and 51.20\% test accuracy.
The best stable streaming operating point we measured is $B=176$ with
\code{stream\_depth=64}$;$ pushing the same shape to
\code{stream\_depth=128} currently runs into shim-DMA descriptor limits. On a
repeated-image benchmark over 100{,}001{,}792 processed images, this
configuration sustains 747{,}851 images/s wall-clock and 3.06 million
images/s if only NPU kernel wait time is counted, corresponding to about
0.20 / 0.81 TFLOPS if the full network forward pass is charged to the
measurement.

The most interesting lesson from this CIFAR-10 point is where the bottleneck
moves. Over that 100M-image run, the host-side $3072\rightarrow32$ embed alone
accounts for about 92.6~s of the 133.7~s wall time, while the NPU kernel itself
accounts for about 32.7~s; the remaining few seconds are other host/runtime
overhead. In other words, once the residual body is narrowed enough and the
batch is pushed high enough, the resident 32-tile snake is no longer the
limiting stage for throughput. The next systems win is not to squeeze a little
more out of the tail worker, but to reduce or eliminate the CPU-side embed
boundary.

Public CIFAR-10 inference-throughput reporting is also much sparser than one
might expect. MLPerf Inference does not currently include a CIFAR-10 task
\cite{mlperf-inference}, so the most relevant public reference points come from
older challenge leaderboards and accelerator-specific papers. Stanford DAWN's
public CIFAR-10 inference leaderboard reports 0.1345~ms/image for a ResNet8
TensorRT submission at 94.20\% accuracy (about 7{,}435 images/s), 0.6830~ms for
a BaiduNet8 V100 submission at 94.32\% accuracy (about 1{,}464 images/s), and
0.8280~ms for a custom ResNet9 P100 submission at 94.53\% accuracy (about
1{,}208 images/s) \cite{dawn-cifar-inference}. FINN reports 21{,}906 CIFAR-10
classifications/s at 80.1\% accuracy on a ZC706 FPGA platform under 25~W total
system power \cite{finn-bnn}. QuTiBench's broader CIFAR-10 benchmark collection
spans Jetson TX2 GPU, Xilinx DPU, FINN, BISMO, Edge TPU-class, and CPU points,
with measured throughput ranging from hundreds of images/s on small edge
deployments to tens of thousands on heavily quantized FPGA configurations
\cite{qutibench-cifar,qutibench-paper}. We deliberately exclude recent A100
CIFAR ``speedrun'' papers such as AIRBench from this comparison because those
results optimize training time to an accuracy target, not steady-state
inference throughput \cite{airbench-cifar-training}.

Against that public inference-only context, the current sustained
747{,}851~images/s wall-clock and 3.06~million images/s kernel-only figures are
strikingly high. The correct caveat, however, is just as important as the
throughput number itself: this $H=32$ operating point reaches only 51.20\% test
accuracy, so it should be read as a systems-throughput result rather than as a
claim that the current model achieves the best known accuracy/throughput
tradeoff on CIFAR-10.

Two follow-up experiments make that limitation much sharper. First, a plain
GPU-trained CIFAR sanity baseline (\code{PreActResNet18}) reaches 90.66\% test
accuracy after 10 epochs on the same machine, so the disappointing CIFAR result
is not a statement that the local stack cannot learn CIFAR-10 at all. It is a
statement about this particular high-throughput residual MLP. Second, a
CIFAR-capable streamed convnet now runs correctly on hardware and matches the
CPU reference numerically, but the correctness-first direct-convolution kernels
collapse throughput to only about 355~images/s wall-clock for a 4-column shape
and about 177~images/s for an 8-column shape, while validation accuracy is
still only about 44.0\% and 35.6\%, respectively. In other words, the current
repository contains two honest but incomplete CIFAR stories: a very fast MLP
that is not yet accurate enough, and a numerically trustworthy conv path that
is not yet remotely competitive on throughput.

That is why CIFAR-10 should currently be read as a useful diagnostic workload,
not as the cleanest headline demo for the resident-snake design. It exposed the
host-side embed bottleneck very clearly, and it also showed that recovering
spatial inductive bias without giving up the conveyor-belt throughput story is a
substantially harder kernel-design problem than retargeting the existing MLP.
Until the faster MMUL-style conv path is recovered, or the surrounding
embed/head stages move onto the array, vector-style workloads remain the more
natural fit for the throughput narrative.

Taken together, these results are still strong while keeping the design
conceptually modest:
\begin{itemize}
  \item no backward pass,
  \item no host weight BO for the residual body,
  \item one checkpoint-specific inference XCLBIN,
  \item one explicit scalar knob, \code{stream\_depth}, controlling the amount
        of in-array overlap,
  \item one repeated shim DMA fill and one repeated shim DMA drain per call,
        rather than one host DMA task per microbatch.
\end{itemize}

That is exactly the kind of systems win worth keeping in the main branch: a
clear design and a measured throughput story. The remaining limitation is now a
much better one to have: not ``can depth 32 work at all?'', but ``how much more
of the surrounding graph should move on-array, and how much closer to peak can
this runtime contract go?''.

\subsection{Literature context for HIGGS}

The HIGGS dataset was introduced by Baldi, Sadowski, and Whiteson as a
large-scale signal-versus-background benchmark for collider physics: roughly
11~million simulated events, a binary label, and 28 floating-point input
features, of which 21 are low-level kinematic variables and 7 are
high-level physics features derived from them \cite{baldi-higgs,uci-higgs}.
That origin matters for this project. Unlike CIFAR-10, HIGGS is already a dense
vector problem, so a residual MLP is at least structurally plausible before any
throughput arguments are made.

The literature around HIGGS then splits into two overlapping traditions. In
high-energy physics, the original result showed that deep networks trained on
the low-level variables can outperform shallower models and reduce the need for
manually engineered high-level features \cite{baldi-higgs}. In general tabular
machine learning, HIGGS became one of the canonical large-data benchmarks for
systems papers on gradient-boosted trees such as XGBoost, LightGBM, and
CatBoost \cite{xgboost-kdd,lightgbm-nips,catboost-paper}. More recent surveys
of tabular learning still conclude that there is no universally superior neural
architecture and that tree ensembles remain the safest overall accuracy
baseline, even as modern tabular deep-learning models become more competitive
\cite{rtdl-revisiting,grinsztajn-tabular}.

One useful modern reference point is the RTDL benchmark of Gorishniy et al.,
which compares tuned deep models and tree ensembles under one protocol on the
reduced \code{higgs\_small} variant \cite{rtdl-revisiting,rtdl-revisit-repo}. The
published test accuracies there are tightly clustered:

Two conclusions from that literature matter for this whitepaper. First, HIGGS
really is a serious benchmark for dense-vector models: modern neural tabular
architectures are competitive there in a way that a plain MLP on CIFAR-10 is
not. Second, the phrase ``state of the art on HIGGS'' has to be handled
carefully. Rankings depend on whether a paper uses the full 11M-row dataset or
the reduced \code{higgs\_small} variant, whether it reports accuracy, AUC, or
physics-specific significance metrics, and how much hyperparameter search is
allowed. For our purposes, the key point is narrower and more practical: HIGGS
is a well-studied vector benchmark on which both boosted trees and modern
tabular neural networks are credible baselines, which makes it a much better
fit for the resident-snake throughput story than a spatial image task.

A useful next question is whether the resident-snake points in this project
stay in that same accuracy band once they are trained on real public data. For
that pilot, I used the public Hugging Face mirror of the HIGGS benchmark splits
\cite{jxie-higgs}, merged \code{train\_63k} and \code{val\_16k} into the local
training pool, kept \code{test\_20k} fixed, and held out 15{,}688 examples
inside the training pool for model selection. Both models were trained for 20
epochs with AdamW ($\text{lr}=3\times10^{-3}$, weight decay $10^{-4}$), a
cosine schedule down to $10^{-4}$, batch size 2048, no label smoothing, and no
data augmentation; the best checkpoints landed at epoch 19 for the $L=8$ model
and epoch 16 for the $L=32$ model. That is not byte-for-byte the exact RTDL
protocol, but it is close enough to test the practical question that matters
here: whether the throughput-friendly residual MLP remains a useful classifier
once the data are real.

not improve the main result. The best current point therefore remains the
throughput-oriented 8-layer configuration at about 4.18M samples/s wall-clock,
and the higher-leverage next step is still to move the $56 \to 32$ embed path
on-array rather than insist on the present fused-head design.

\section{Toolchain in one page}

The software pipeline in this repository is short enough to summarize directly:
\begin{lstlisting}
aie_kernels/matmul_relu_skip.cc
    -> LLVM-AIE object
    -> kernel archive

resmlp/design.py
    -> Python-generated MLIR-AIE module

MLIR-AIE + aiecc
    -> .xclbin + instruction binary

XRT runtime
    -> buffer setup + runlist execution on the NPU
\end{lstlisting}

The IRON project and the MLIR-AIE ecosystem are the main external dependencies
for the hardware side of this repository \cite{iron,mlir-aie}. The Linux AMD NPU
documentation is also useful for understanding the runtime and driver-facing view
of the device \cite{amd-npu-linux}.
