\large\color{black!70} High-throughput residual inference for HIGGS on AMD XDNA~2
\end{center}

\begin{abstract}
We study a deliberately narrow question: which neural-network shape best
matches the AMD XDNA~2 NPU for high-throughput, low-latency inference on dense
collider-style features? We answer it with a forward-only residual MLP whose
hidden body streams across the 32 compute tiles of the Strix Point array. On
the public HIGGS benchmark, we retain three key configurations: a host-assisted
$H=32$, $L=8$ model that sustains about 2.38M samples/s wall-clock, a retrained
fully on-array $H=32$, $L=30$ model that reaches 76.61\% test accuracy and
14.08M samples/s wall-clock at $B=64$ (2.44M at the native streaming batch
$B=8$), and a full-data $H=64$, $L=32$ model trained on 10.4M examples that
reaches 77.98\% test accuracy, 0.865 ROC AUC, and 0.877 PR AUC on the fixed
500k-example test split. The design is small enough to understand end to end
yet strong enough to support a credible systems case for dense event filtering
on consumer hardware. The contribution is a measured co-designed operating
regime, not a production trigger deployment, a state-of-the-art HIGGS accuracy
paper, or an energy-efficiency study.
\end{abstract}

\section{Introduction}

Modern NPUs become most interesting when they stop being treated as ``small
GPUs'' and start being programmed as spatial dataflow machines. Instead of
launching one large shared-memory kernel, the programmer assigns a small local
job to one tile, hands the activations to the next tile, and repeats until the
full network has been traversed. On AMD XDNA~2, the programmer sees a 2D array
of compute tiles, each with local SRAM, a VLIW plus SIMD core, a
matrix-multiply unit, and explicit routing. The central question is therefore
no longer just ``what architecture is accurate?'', but also ``what architecture
maps cleanly to many tiny memories connected by explicit streams?''.

The focus here is deliberately narrow: a residual MLP for HIGGS, with one
residual matrix per tile and a forward-only spatial-streaming deployment on the
NPU. The aim is not to survey every model family or every use of XDNA~2, but
to isolate one design that can be understood end to end.

HIGGS is a natural benchmark for this study. It is a dense 28-feature binary
classification task derived from collider events, introduced by Baldi,
Sadowski, and Whiteson as a large-scale signal-versus-background benchmark with
roughly 11 million examples \cite{baldi-higgs,uci-higgs}. That makes a residual
MLP structurally plausible before any hardware argument is made. It also makes
the throughput story easier to defend. For example, the CMS Phase-2 Level-1
trigger must inspect detector data at the 40~MHz bunch-crossing rate, produce
decisions with a latency budget of about 12.5~$\mu$s, and emit accepted events
at roughly 750~kHz \cite{cms-phase2-l1}. Those rates explain why
multi-megasample-per-second measurements are interesting here, but they are not
used to imply deployment equivalence: the best wall-clock repeated-sample
result in this paper is 14.08M samples/s at $B=64$ on a fully on-array
$H=32$, $L=30$ path, while the same model reaches 2.44M samples/s at the native
$B=8$ streaming batch. Those figures sit above the accepted-event rate yet far
below the full bunch-crossing stream. The point is to study the same general
systems regime of dense features, strict latency sensitivity, and pressure for
very high sustained throughput.

This places the paper next to three adjacent literatures. For HIGGS itself, the
reference landscape already includes both the original deep-versus-classic
baseline framing and the strong tabular baselines now typically led by boosted
tree ensembles \cite{baldi-higgs,xgboost-kdd,lightgbm-nips,catboost-paper,rtdl-revisiting,grinsztajn-tabular}; attention-based and decision-tree-hybrid deep tabular methods such as TabNet and NODE have also been evaluated on this benchmark \cite{tabnet,node-paper}. Comprehensive surveys of machine learning for particle physics provide broader context for where such dense-event classification sits within the HEP community \cite{ml4hep-review}. For
low-latency collider inference, substantial prior work has targeted FPGA-based
trigger deployments and related co-design flows; the hls4ml open codesign
toolchain makes the path from a trained model to firmware particularly direct
\cite{duarte-fpga,hls4ml,cms-phase2-l1}. At
the accelerator level, the closest precedent is the spatial/dataflow tradition,
where local SRAM and explicit movement constrain the network shape as much as
raw arithmetic throughput does; purpose-built DNN accelerators and the Google
TPU family both illustrate how data movement can become the primary design
constraint rather than arithmetic throughput \cite{dadiannao,eyeriss,tpu-isca17}. The claim here is
narrower: given an existing benchmark, existing consumer NPU hardware, and an
existing compiler/runtime stack, what is the simplest residual design that maps
cleanly enough to the array to make a credible end-to-end systems case?

These constraints shape the model directly. The hidden body is kept square so
that each tile stores one $H \times H$ residual weight matrix. Widths are kept
as multiples of 8 to match the AIE MMUL datapath. The host only moves the data
that are still cheaper to keep on CPU today: the input projection and, for the
fastest retained variant, the tiny final $H \rightarrow 2$ classifier head.

\section{A residual network that matches the array}

The network studied here is intentionally minimal.

\subsection*{Notation}

We use $H$ for hidden width, $L$ for the number of residual blocks, and $B$ for
batch size. In the deployment path studied here, one residual block maps to one
compute tile. We use $\rightarrow$ for layer mappings (for example
$H \rightarrow 2$ classifier) and $\times$ for matrix shapes (for example
$H \times H$ weights).

\subsection*{Model definition}

\[
  h_{\ell+1} = \mathrm{ReLU}(h_{\ell} W_{\ell}) + h_{\ell},
  \qquad \ell = 0,\dots,L-1.
\]
The skip connection follows the convention introduced by He et al.~\cite{he-resnet};
each residual matrix $W_{\ell}$ is assigned to one compute tile. Activations
enter the array, flow through the tiles in a snake, and return to the host only
after the full hidden body has completed. In the current deployment path:
\begin{itemize}
  \item the host CPU runs the input projection from the native 28 HIGGS features
         to hidden width $H$,
  \item the NPU runs the residual body,
  \item the host CPU currently keeps the tiny final $H \rightarrow 2$ head when
         that is faster than the available fused on-array tail.
\end{itemize}

This design leads to two retained operating variants.
\begin{itemize}
  \item \textbf{Speed-optimized variant:} $H=32$, $L=8$, mapped to 2 columns. This is
         the fastest end-to-end HIGGS point measured so far.
  \item \textbf{Accuracy-tuned variant:} $H=64$, $L=32$, trained on the full public
         dataset and mapped to the full 8-column array when the deeper body is
         streamed on the NPU.
\end{itemize}

The retained paper path keeps the narrow surround on CPU and uses the NPU only
for the residual streaming body. The same snake-order placement then scales
from the 2-column $L=8$ speed-optimized variant to the full 8-column $L=32$
body; Figure~\ref{fig:model-hardware-mapping} makes that deployment contract
concrete.

The important design choice is not just ``use an MLP'', but ``use an MLP whose
depth and width map directly to the array's tile structure''. That keeps the design
simple enough to analyze and direct enough to deploy.

\section{From Python model to NPU artifact}

The toolchain is deliberately short.
\path{aie_kernels/matmul_relu_skip.cc} defines the tile-local residual kernel.
\path{resmlp/streaming_design.py} turns a chosen $(H, B, L)$ shape into a
spatial-streaming MLIR-AIE program. \path{resmlp/streaming_op.py} wraps the
compiled XCLBIN and instruction binary as a resident XRT operator. Finally,
\path{resmlp/streaming_infer.py} drives trained checkpoints through that
operator and measures wall-clock throughput.

The novelty claimed here is not a new compiler or runtime. IRON and MLIR-AIE
already provide the compilation substrate \cite{iron,mlir-aie}; what is new is
the model-to-array mapping, the host/NPU partition, and the end-to-end
measurements of that co-designed path on real HIGGS checkpoints.

A key detail of the current forward path is that the residual weights are
embedded at compile time. Each throughput run therefore builds a
checkpoint-specific streaming artifact. That is not the most general
deployment contract. It is, however, the cleanest way to study the geometry,
routing, and wall-clock behavior of a resident spatial-streaming design.

\section{Training protocol on HIGGS}

The data pipeline uses the public \code{jxie/higgs} mirror of the published
train/test splits \cite{jxie-higgs}. The local cache materializer produces a
fixed 10.5M-example train split and a fixed 500k-example test split. For model
selection, training holds out 100k examples from the train split. That leaves
10.4M examples for optimization. All reported runs use the native
28-dimensional feature vector without any padded-width preprocessing.
Hyperparameter search and checkpoint selection use held-out validation
accuracy. The 500k-example test split is used only for final evaluation of
already-selected models.

HIGGS is also a good benchmark because it sits at the intersection of two real
literatures. In high-energy physics it is a canonical dense-event benchmark
\cite{baldi-higgs}, situated within a broader programme of applying machine
learning to particle-physics classification and reconstruction that is
surveyed in \cite{ml4hep-review}. In tabular machine learning it is one of the standard
large-data problems on which tree ensembles such as XGBoost, LightGBM, and
CatBoost remain strong baselines, while modern deep tabular models---including
attention-based and decision-tree-hybrid approaches---are now more
competitive but not universally dominant \cite{xgboost-kdd,lightgbm-nips,catboost-paper,rtdl-revisiting,grinsztajn-tabular,tabnet,node-paper}.
That means the relevant bar is not ``beat every HIGGS leaderboard'', but
``land in a statistically credible part of the baseline landscape while
preserving a deployment shape the NPU can actually exploit''. Good-enough
accuracy on HIGGS is therefore genuinely worth caring about even if the main
contribution of this work remains systems-oriented.

\subsubsection{Reducing host-loop overhead with a double-buffered host loop}
\label{sec:pipeline-overlap}

Earlier revisions of this paper reported that wall throughput was several times
lower than kernel throughput for every configuration. Instrumenting the host
loop showed that the gap was dominated by Python per-call overhead: the
$H=64$, $L=32$ configuration spent roughly 380~\textmu{}s/call on the CPU
input projection, 64~\textmu{}s/call tiling the activations, 78~\textmu{}s/call
detiling outputs, and 71~\textmu{}s/call on the CPU head, against only
156~\textmu{}s/call in kernel wait. Two changes attack this gap without
changing the NPU kernel or the network itself. First, we replaced 32 small
Python dispatches per kernel invocation with four stacked numpy operations
(one embed matmul on all $S \!\times\! B$ rows, one reshape/transpose pack, one
reshape/transpose detile, one head matmul), which collapses the host work
from ${\sim}590$\,\textmu{}s/call to roughly 116~\textmu{}s/call on the
$H=64$, $L=32$ checkpoint. Second, we use two-slot ping-pong device buffers and
dispatch the next kernel with \texttt{run = xrt\_kernel(\dots{})} before the
host begins consuming the previous slot's output, so a large fraction of the
host work overlaps with kernel execution instead of paying the full
$\text{host}+\text{kernel}$ sum on every iteration.

The corrected measurements now show four different regimes. For the two hybrid
$H=32$ variants the async host loop materially improves wall throughput over the
serial baseline and nearly closes the gap to isolated kernel throughput: the
$L=8$ speed variant rises from 1.72M to 2.38M samples/s and lands at 97\% of
isolated kernel throughput, while the deeper $H=32$, $L=32$ variant rises from
1.64M to 2.25M and lands at 99\%. The retrained full-NPU $H=32$, $L=30$ path
goes one step further by eliminating the CPU embed/head matmuls entirely. At
$B=8$ it reaches 2.44M samples/s wall-clock and 99\% of isolated kernel
throughput, a modest but real gain over the host-assisted $H=32$, $L=32$
baseline. At $B=64$, the same fully on-array graph scales to 14.08M samples/s
wall-clock while still landing at 99.6\% of isolated kernel throughput. The
wider $H=64$, $L=32$ accuracy variant also improves substantially---from 0.82M
to 1.45M samples/s ($+78\%$)---but still reaches only 73\% of isolated kernel
throughput. There the host path remains a real wall-time cost: the benchmark
reports about 72~\textmu{}s/call in pre-dispatch host work and 43~\textmu{}s/call
in post-wait host work against a 128~\textmu{}s/call isolated kernel.

This sharpens the remaining ``NPU-native'' question rather than eliminating it.
For narrow $H=32$ bodies, moving the input projection and tiny head onto the
array is now shown to help in two distinct ways: it removes most of the
host-visible work at $B=8$ and opens a much steeper scaling curve at larger
batches, where the fully on-array path reaches 14.08M samples/s. For the wider
$H=64$ configuration, however, host-side work still consumes a substantial
fraction of the wall path, so further host/runtime optimization or a cleaner
on-array ingress/egress path for that wider regime could still buy real
throughput.

Figure~\ref{fig:accuracy-throughput-frontier} combines the retained full-data
accuracy rows with the measured async wall-throughput rows. It makes the
trade-off explicit: the retrained full-NPU $H=32$, $L=30$ path is now the

\subsection{CPU throughput baseline}

To put the NPU throughput numbers in context, we measure the same models on a
laptop CPU using both PyTorch and a decomposed NumPy path that mirrors the
NPU's inference structure (separate embed, sequential residual body, head).

\paragraph{Hardware and software.}
The baseline machine is the same AMD Ryzen AI 9 HX 370 (12 cores / 24 threads,
max 5.16~GHz) used for all NPU measurements, running Linux 6.17. PyTorch
2.10.0 uses 12 intra-op threads (the physical core count); NumPy 1.26.4 is
backed by OpenBLAS with the same default thread pool. No explicit thread pinning
or frequency governor tuning was applied; the measurements reflect the default
out-of-the-box configuration.

\paragraph{Methodology.}
Each benchmark processes 5M samples using random \code{float32} inputs of the
correct dimensionality ($D=28$). Ten warmup calls precede the timed region.
The PyTorch path calls \code{model.forward()} under \code{torch.no\_grad()};
the NumPy path applies the embed, residual chain, and head as explicit
\code{numpy.matmul} calls, matching the host/NPU split used in the streaming
deployment. We sweep batch sizes from $B=8$ (matching the NPU's streaming batch)
through $B=1024$ to show how CPU throughput scales with batch-level parallelism.

Several patterns are worth noting. First, the CPU strongly favors large batch
sizes: PyTorch throughput on the speed variant improves by $\sim31\times$ going
from $B=8$ to $B=1024$, because larger batches amortize per-call dispatch
overhead and expose enough work for BLAS multi-threading. At $B=8$, the
$32\times32$ matmuls are too small for OpenBLAS to distribute across cores, and
PyTorch's per-layer dispatch adds measurable overhead; at larger batch sizes,
OpenBLAS saturates the 12 available threads and PyTorch's fused ATen kernels
pay off. Second, the NumPy decomposed path is faster than PyTorch at small batch
sizes (lower per-call overhead) but slower at large batch sizes where PyTorch's
internal parallelism dominates.

Third, and most relevant to the NPU comparison: at the NPU's native streaming
batch size of $B=8$, the CPU reaches only 104k~samples/s (PyTorch) for the
speed variant---roughly $23\times$ slower than the hybrid NPU path's
2.38M~samples/s wall-clock throughput and $23.5\times$ slower than the retrained
fully on-array path's 2.44M~samples/s. Even at the CPU's best batch size
($B=1024$), the peak of 3.27M samples/s is still far below the full-NPU
$B=64$ result of 14.08M~samples/s. For the accuracy variant ($H=64$, $L=32$),
the NPU retains a clear advantage: the CPU peaks at 598k~samples/s versus the
NPU's 1.45M~samples/s wall-clock, a $2.4\times$ factor.

\paragraph{Note.}
Both the CPU baseline and the NPU throughput measurements were obtained on the
same AMD Ryzen AI 9 HX 370 host, making the comparison controlled: the two
results share the same memory subsystem, OS, and software stack, differing only
in whether the residual body runs on the host cores or the on-die NPU array.

\section{Discussion}

The claim is deliberately narrow but defensible.
\begin{itemize}
  \item A residual MLP can be shaped to match the AMD XDNA~2 array very
        directly: one residual matrix per tile, streamed activations, explicit
        routing, and checkpoint-specific resident weights.
  \item That design already delivers multi-million-sample-per-second inference on
        a real collider-style dense benchmark using cheap consumer hardware,
        culminating in 14.08M samples/s on a retrained fully on-array
        $H=32$, $L=30$ path.
  \item The same design family also reaches nontrivial full-data HIGGS quality,
        culminating in 77.98\% test accuracy and 0.865 ROC AUC on the fixed
        500k-example test set.
\end{itemize}

Just as importantly, the paper avoids claims it does not need. It does not
present the current model as a production trigger deployment, the current HIGGS
result as state of the art, or the present measurements as an energy-efficiency
study. The benchmark is
public, the hardware is existing AMD XDNA~2, and the compiler/runtime substrate
is pre-existing; none of those ingredients are introduced here. The novelty is
the co-designed operating regime: a residual MLP whose hidden-body geometry
matches the array closely enough that explicit spatial deployment yields both a
useful throughput result and a statistically credible HIGGS checkpoint.

A useful boundary from the SRAM accounting is that wider hidden bodies and
wider surrounding inputs do not stress the current deployment in the same way.
For the present $B=8$ spatial-streaming layout, an internal residual tile carries
roughly $1024 + 2H^2 + 32H$ bytes, while the ingress tile carries
$1024 + 2H^2 + 64H$ because it must hold both the host-ingress buffers and the
first residual stage. That makes the first tile, not the middle of the belt,
the width bottleneck. The retained layout still fits at $H=160$ with about
62.5~KB live on the ingress tile, but the same accounting already pushes
$H=168$ past 64~KB there even though the internal tiles would still fit. In the
current paper, then, widening the hidden body is a genuine tile-SRAM limit of
the current spatial-streaming placement.

The surrounding stages are different. For the native 28-feature HIGGS input,
even a padded $32 \rightarrow H$ input projection carries only $64H$ bytes of
bf16 weights (4~KB at $H=64$, 10~KB at $H=160$), so keeping that stage on CPU
today should not be read as evidence of an SRAM bottleneck. The awkward
part is that 28 inputs do not naturally match the 8$\times$8 MMUL granularity,
so an on-array ingress path must pay packing, alignment, and boundary-management
costs. The same distinction is even clearer for the tiny $H \rightarrow 2$
head: its weight footprint is negligible, and the measured slowdown of the
fused NPU head is best understood as synchronization and launch overhead. This
keeps the scope narrow. Within the HIGGS-style feature regime studied here,
wider surrounding inputs mainly stress alignment and runtime boundaries,
whereas the hard SRAM wall comes from widening the hidden body under the
present first-tile layout.

\section{Conclusion and next steps}

The main result is simple: a network designed for spatial compute on AMD
XDNA~2 can deliver unusually high throughput on real dense-event
classification while remaining small enough to understand and validate end to
end. The model family is deliberately modest, but it maps cleanly to the array
and now yields three concrete operating points: a host-assisted speed path,
a fully on-array $H=32$, $L=30$ path that reaches 14.08M~samples/s, and a
nontrivial full-data $H=64$, $L=32$ HIGGS result.

What exists today is a measured prototype family spanning both host-assisted and
fully on-array inference paths; energy-efficiency characterization remains
future work.

The next steps are correspondingly focused. The double-buffered host loop
(\S\ref{sec:pipeline-overlap}) already removes a large fraction of the earlier
serial host overhead and materially improves end-to-end wall throughput. The
new measurements now show a split outcome: for narrow $H=32$ bodies, a
fully on-array ingress/egress path is worthwhile and scales especially well at
larger batches, while the wider $H=64$ accuracy model still leaves substantial
wall time on the host side and has not yet been retrained in a corresponding
full-NPU form. In parallel, genuine kernel-side gains remain valuable:
increasing stream depth up to the per-DMA descriptor limit, overlapping the
D2H of one chunk with the H2D of the next, or revisiting the MMUL tiling so
that more compute is packed into each kernel invocation. The SRAM-sensitive
scaling question is different: wider hidden bodies will require revisiting the
first-tile ingress layout because that tile reaches the 64~KB ceiling before
the internal tiles do. The highest-leverage next modeling step is to test
whether the full-NPU ingress/egress strategy that works well for $H=32$ can be
extended profitably to the more accurate $H=64$, $L=32$ regime.

\section{HIGGS physics background}

The HIGGS benchmark comes from a standard collider-analysis setting: a rare
target process must be separated from much more common look-alike processes
using reconstructed event kinematics \cite{baldi-higgs}. The label is binary.
In the UCI release, `1' denotes signal and `0' denotes background
\cite{uci-higgs}. Physically, that does not mean the detector has directly
``seen a Higgs.'' It means the reconstructed pattern of visible objects in the
event is more consistent with the target signal process than with ordinary
background production.

Each row is therefore already one reconstructed collision event, not raw
detector data. The low-level variables describe the visible ingredients of the
event: one charged lepton, missing transverse momentum, and up to four jets,
plus per-jet b-tag indicators \cite{uci-higgs}. In collider language, $p_T$ is
momentum transverse to the beam, $\eta$ and $\phi$ encode direction, and
missing transverse momentum records the momentum imbalance associated with
invisible particles leaving the detector. The high-level variables are
invariant-mass combinations such as $m_{jj}$, $m_{jjj}$, $m_{lv}$, $m_{jlv}$,
$m_{bb}$, $m_{wbb}$, and $m_{wwbb}$; they test whether groups of reconstructed
objects line up with a plausible underlying resonance or decay chain
\cite{uci-higgs}.

That structure is exactly why HIGGS is useful here. The model does not have to
solve raw reconstruction. It receives a compact physics summary of each event
and must make a fast signal-versus-background decision on dense numeric inputs.
In this repository those examples come from the public \code{jxie/higgs} mirror
of the published train/test split \cite{jxie-higgs}.

\centering
\includegraphics[width=\textwidth]{higgs_task_diagram.pdf}
\caption{From reconstructed collision event to binary label in HIGGS. The
dataset stores lepton, jet, missing-momentum, and derived invariant-mass
features; the classifier must separate rare signal-like events from common
