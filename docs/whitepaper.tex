\begin{center}
\large Conveyor-belt residual inference for HIGGS on AMD XDNA~2
\end{center}

\begin{abstract}
This paper studies a deliberately narrow question: what neural-network shape
matches the AMD XDNA~2 NPU when the goal is high-throughput, low-latency
inference on dense collider-style features? The answer explored here is a
forward-only residual MLP whose hidden body is laid out as a conveyor belt
across the 32 compute tiles of the Strix Point array. Using the public HIGGS
benchmark, we report two operating points: a throughput-oriented $H=32$,
$L=8$ configuration that sustains about 4.18M samples/s wall-clock with the
tiny classifier head left on CPU, and a full-data $H=64$, $L=32$ model trained
on 10.4M examples that reaches 77.98\% test accuracy, 0.865 ROC AUC, and 0.877
PR AUC on the fixed 500k-example test split. The point is not that this model
already wins every accuracy leaderboard; it is that the design is small enough
to understand end to end, yet strong enough to make a credible systems case for
dense event filtering on cheap consumer hardware.
\end{abstract}

\section{Introduction}

Modern NPUs become most interesting when they stop being treated as ``small
GPUs'' and start being treated more like conveyor belts. Instead of launching
one large shared-memory kernel, the programmer assigns a small local job to one
tile, hands the activations to the next tile, and repeats this until the full
network has been traversed. That is the intuition behind spatial dataflow
hardware. On AMD XDNA~2, the programmer sees a 2D array of compute tiles, each
with local SRAM, a VLIW plus SIMD core, a matrix-multiply unit, and explicit
routing. The central question is therefore no longer just ``what architecture
is accurate?'', but also ``what architecture maps cleanly to many tiny memories
connected by explicit streams?''.

The focus here is deliberately narrow: a residual MLP for HIGGS, with one
residual matrix per tile and a forward-only conveyor-belt deployment on the
NPU. The aim is not to survey every model family or every use of XDNA~2, but
to isolate one design that can be understood end to end.

HIGGS is a natural benchmark for this study. It is a dense 28-feature binary
classification task derived from collider events, introduced by Baldi,
Sadowski, and Whiteson as a large-scale signal-versus-background benchmark with
roughly 11 million examples \cite{baldi-higgs,uci-higgs}. That makes a residual
MLP structurally plausible before any hardware argument is made. It also makes
the throughput story easier to defend. For example, the CMS Phase-2 Level-1
trigger must inspect detector data at the 40~MHz bunch-crossing rate, produce
decisions with a latency budget of about 12.5~$\mu$s, and emit accepted events
at roughly 750~kHz \cite{cms-phase2-l1}. This project is not a production
trigger design, but it does live in the same systems regime: dense features,
strict latency sensitivity, and strong pressure for very high sustained
throughput.

These constraints shape the model directly. The hidden body is kept square so
that each tile stores one $H \times H$ residual weight matrix. Widths are kept
as multiples of 8 to match the AIE MMUL datapath. The host only moves the data
that are still cheaper to keep on CPU today: the input embed and, for the best
current throughput point, the tiny final $H \rightarrow 2$ classifier head.

\section{A residual network that matches the array}

The network studied here is intentionally minimal.

\subsection*{Notation}

We use $H$ for hidden width, $L$ for the number of residual blocks, and $B$ for
batch size. In the deployment path studied here, one residual block maps to one
compute tile.

\subsection*{Model definition}

\[
  h_{\ell+1} = \mathrm{ReLU}(h_{\ell} W_{\ell}) + h_{\ell},
  \qquad \ell = 0,\dots,L-1.
\]
Each residual matrix $W_{\ell}$ is assigned to one compute tile. Activations
enter the array, flow through the tiles in a snake, and return to the host only
after the full hidden body has completed. In the current deployment path:
\begin{itemize}
  \item the host CPU runs the input projection from the native 28 HIGGS features
        to hidden width $H$,
  \item the NPU runs the residual body,
  \item the host CPU currently keeps the tiny final $H \rightarrow 2$ head when
        that is faster than the available fused on-array tail.
\end{itemize}

This design leads to two practical operating points.
\begin{itemize}
  \item \textbf{Throughput point:} $H=32$, $L=8$, mapped to 2 columns. This is
        the fastest end-to-end HIGGS point measured so far.
  \item \textbf{Accuracy point:} $H=64$, $L=32$, trained on the full public
        dataset and mapped to the full 8-column array when the deeper body is
        streamed on the NPU.
\end{itemize}

The important design choice is not just ``use an MLP'', but ``use an MLP whose
depth and width map almost literally to the array''. That keeps the design
simple enough to analyze and direct enough to deploy.

\section{From Python model to NPU artifact}

The toolchain is deliberately short. We start from
\code{aie\_kernels/matmul\_relu\_skip.cc}, which defines the tile-local
residual kernel. We then use \code{resmlp/streaming\_design.py} to generate the
conveyor-belt MLIR-AIE program for a chosen $(H, B, L)$ shape.
\code{resmlp/streaming\_op.py} wraps the compiled XCLBIN and instruction binary
as a resident XRT operator. Finally, \code{resmlp/streaming\_infer.py} drives
trained checkpoints through that operator and measures wall-clock throughput.

A key detail of the current forward path is that the residual weights are
embedded at compile time. Each throughput run therefore builds a
checkpoint-specific streaming artifact. That is not the most general
deployment contract. It is, however, the cleanest way to study the geometry,
routing, and wall-clock behavior of a resident conveyor-belt design.

\section{Training protocol on HIGGS}

The data pipeline uses the public \code{jxie/higgs} mirror of the published
train/test splits \cite{jxie-higgs}. The local cache materializer produces a
fixed 10.5M-example train split and a fixed 500k-example test split. For model
selection, training holds out 100k examples from the train split. That leaves
10.4M examples for optimization. All reported runs use the native
28-dimensional feature vector without any padded-width preprocessing.
Hyperparameter search and checkpoint selection use held-out validation
accuracy. The 500k-example test split is used only for final evaluation of
already-selected models.

HIGGS is also a good benchmark because it sits at the intersection of two real
literatures. In high-energy physics it is a canonical dense-event benchmark
\cite{baldi-higgs}. In tabular machine learning it is one of the standard
large-data problems on which tree ensembles such as XGBoost, LightGBM, and
CatBoost remain strong baselines, while modern deep tabular models are now more
competitive but not universally dominant \cite{xgboost-kdd,lightgbm-nips,catboost-paper,rtdl-revisiting,grinsztajn-tabular}.
That means good-enough accuracy on HIGGS is genuinely worth caring about, even
if the main contribution of this work is still systems-oriented.

\section{Results}

\subsection{Accuracy on the full public HIGGS split}

Table~\ref{tab:higgs-full-results} summarizes the current full-data results.
The key point is that the systems design is not accuracy-indifferent. Even this
deliberately hardware-friendly residual MLP becomes a credible classifier once
it is trained on the full public HIGGS data.

Two observations matter. First, depth helps. Moving from $L=8$ to $L=32$
improves every metric even before any serious hyperparameter search. Second,
validation-driven hyperparameter tuning pushes the result from the high-0.85
ROC band to 0.865 ROC AUC. It also improves accuracy, PR AUC, and log loss.
This is still not a claim of state-of-the-art HIGGS accuracy. It is, however,
strong enough that the main contribution of the paper is no longer
accuracy-indifferent.

This table sharpens the systems conclusion. The best current wall-clock point
is not the deepest model, and it is not the version with the entire graph moved
onto the array. On the present tail kernel, the fused $H \rightarrow 2$ NPU head
is slower than keeping that tiny classifier on CPU. That is an important honest
result: the right abstraction boundary in the current implementation is a CPU
embed, NPU residual body, and CPU head. The next hardware step is therefore not
``force everything on-array'' but ``move the expensive surrounding stages only
when they really improve wall time''. A plausible reason is that the
$H \rightarrow 2$ head is simply too small to amortize the extra synchronization,
buffer traffic, and launch overheads of the fused tail path. In other words,
the additional runtime machinery costs more than the CPU saves on this tiny
matrix multiply.

\section{Discussion}

This study makes a narrow but defensible claim.
\begin{itemize}
  \item A residual MLP can be shaped to match the AMD XDNA~2 array very
        directly: one residual matrix per tile, streamed activations, explicit
        routing, and checkpoint-specific resident weights.
  \item That design already delivers multi-million-sample-per-second inference on
        a real collider-style dense benchmark using cheap consumer hardware.
  \item The same design family also reaches nontrivial full-data HIGGS quality,
        culminating in 77.98\% test accuracy and 0.865 ROC AUC on the fixed
        500k-example test set.
\end{itemize}

Just as importantly, the paper avoids claims it does not need. It does not
pretend the current model is a production trigger, that the current tail kernel
is optimal, or that the current HIGGS result is the final word on tabular
accuracy. The point is simpler and cleaner: spatial hardware rewards spatially
shaped models, and a small, explicit conveyor-belt design can already hit a
useful combination of interpretability, throughput, and real-task credibility.

\section{Conclusion and next steps}

The main result is straightforward: designing a network for spatial compute on
AMD XDNA~2 can produce unusually high throughput on real dense-event
classification while remaining simple enough to understand and validate end to
end. The model family is deliberately modest, but it maps cleanly to the array
and delivers both a strong throughput point and a nontrivial full-data HIGGS
accuracy point.

The next steps are correspondingly focused. The highest-leverage systems step is
to move the $28 \rightarrow 32$ embed on-array so that the surrounding host work
shrinks further. The highest-leverage modeling step is to continue exploiting
the promising $H=64$, $L=32$ region with longer schedules and regularization,
rather than broadening the scope of the architecture prematurely.
