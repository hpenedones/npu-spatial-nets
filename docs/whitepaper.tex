\begin{center}
\large\color{black!70} High-throughput residual inference for HIGGS on AMD XDNA~2
\end{center}

\begin{abstract}
We study a deliberately narrow question: which neural-network shape best
matches the AMD XDNA~2 NPU for high-throughput, low-latency inference on dense
collider-style features? We answer it with a forward-only residual MLP whose
input projection, 30 residual blocks, and classifier head all stream across the
32 compute tiles of the Strix Point array. On the public HIGGS benchmark, this
fully on-array $H=32$, $L=30$ pipeline reaches 76.61\% test accuracy and
sustains 14.08M samples/s wall-clock at batch size 64 on the fixed 500k-example
test split, or 2.44M samples/s at the native streaming batch of 8. On the same
machine and the same model, a 12-core CPU baseline peaks at 0.98M samples/s,
so the on-array path is roughly $14\times$ faster at the best CPU operating
point. The design is
small enough to understand end to end yet strong enough to support a credible
systems case for dense event filtering on consumer hardware. The contribution
is a measured co-designed operating regime, not a production trigger
deployment, a state-of-the-art HIGGS accuracy paper, or an energy-efficiency
study.
\end{abstract}

\section{Introduction}

Modern NPUs become most interesting when they stop being treated as ``small
GPUs'' and start being programmed as spatial dataflow machines. Instead of
launching one large shared-memory kernel, the programmer assigns a small local
job to one tile, hands the activations to the next tile, and repeats until the
full network has been traversed. On AMD XDNA~2, the programmer sees a 2D array
of compute tiles, each with local SRAM, a VLIW plus SIMD core, a
matrix-multiply unit, and explicit routing. The central question is therefore
no longer just ``what architecture is accurate?'', but also ``what architecture
maps cleanly to many tiny memories connected by explicit streams?''.

The focus here is deliberately narrow: a residual MLP for HIGGS, with one
residual matrix per tile and a forward-only spatial-streaming deployment on the
NPU. The aim is not to survey every model family or every use of XDNA~2, but
to isolate one design that can be understood end to end.

HIGGS is a natural benchmark for this study. It is a dense 28-feature binary
classification task derived from collider events, introduced by Baldi,
Sadowski, and Whiteson as a large-scale signal-versus-background benchmark with
roughly 11 million examples \cite{baldi-higgs,uci-higgs}. That makes a residual
MLP structurally plausible before any hardware argument is made. It also makes
the throughput story easier to defend. For example, the CMS Phase-2 Level-1
trigger must inspect detector data at the 40~MHz bunch-crossing rate, produce
decisions with a latency budget of about 12.5~$\mu$s, and emit accepted events
at roughly 750~kHz \cite{cms-phase2-l1}. Those rates explain why
multi-megasample-per-second measurements are interesting here, but they are not
used to imply deployment equivalence: the wall-clock repeated-sample result in
this paper is 14.08M samples/s at $B=64$ on a fully on-array $H=32$, $L=30$
path, which sits above the accepted-event rate yet far below the full
bunch-crossing stream. The point is to study the same general systems regime
of dense features, strict latency sensitivity, and pressure for very high
sustained throughput.

This places the paper next to three adjacent literatures. For HIGGS itself, the
reference landscape already includes both the original deep-versus-classic
baseline framing and the strong tabular baselines now typically led by boosted
tree ensembles \cite{baldi-higgs,xgboost-kdd,lightgbm-nips,catboost-paper,rtdl-revisiting,grinsztajn-tabular}; attention-based and decision-tree-hybrid deep tabular methods such as TabNet and NODE have also been evaluated on this benchmark \cite{tabnet,node-paper}. Comprehensive surveys of machine learning for particle physics provide broader context for where such dense-event classification sits within the HEP community \cite{ml4hep-review}. For
low-latency collider inference, substantial prior work has targeted FPGA-based
trigger deployments and related co-design flows; the hls4ml open codesign
toolchain makes the path from a trained model to firmware particularly direct
\cite{duarte-fpga,hls4ml,cms-phase2-l1}. At
the accelerator level, the closest precedent is the spatial/dataflow tradition,
where local SRAM and explicit movement constrain the network shape as much as
raw arithmetic throughput does; purpose-built DNN accelerators and the Google
TPU family both illustrate how data movement can become the primary design
constraint rather than arithmetic throughput \cite{dadiannao,eyeriss,tpu-isca17}. The claim here is
narrower: given an existing benchmark, existing consumer NPU hardware, and an
existing compiler/runtime stack, what is the simplest residual design that maps
cleanly enough to the array to make a credible end-to-end systems case?

These constraints shape the model directly. The hidden body is kept square so
that each tile stores one $H \times H$ residual weight matrix. Widths are kept
as multiples of 8 to match the AIE MMUL datapath. The retained deployment
places the entire inference path on the array: the first tile runs a padded
$32 \rightarrow H$ embed, the next 30 tiles run the residual body, and the last
tile runs a padded $H \rightarrow 8$ head, leaving only small pad/strip
operations on the host.

\section{A residual network that matches the array}

The network studied here is intentionally minimal.

\subsection*{Notation}

We use $H$ for hidden width, $L$ for the number of residual blocks, and $B$ for
batch size. In the deployment path studied here, one residual block maps to one
compute tile. We use $\rightarrow$ for layer mappings (for example
$H \rightarrow 2$ classifier) and $\times$ for matrix shapes (for example
$H \times H$ weights).

\subsection*{Model definition}

\[
  h_{\ell+1} = \mathrm{ReLU}(h_{\ell} W_{\ell}) + h_{\ell},
  \qquad \ell = 0,\dots,L-1.
\]
The skip connection follows the convention introduced by He et al.~\cite{he-resnet};
each residual matrix $W_{\ell}$ is assigned to one compute tile. Activations
enter the array, flow through the tiles in a snake, and return to the host only
after the full pipeline has completed. In the retained deployment path:
\begin{itemize}
  \item the first compute tile runs a padded $32 \rightarrow H$ input projection
        that maps native 28-feature HIGGS events into the hidden width,
  \item the next 30 tiles run the residual body, one $H \times H$ matrix per
        tile,
  \item the last tile runs a padded $H \rightarrow 8$ classifier head whose
        first two output columns carry the HIGGS binary logits.
\end{itemize}
With $H=32$ this fills the full 32-tile array. The host only pads the 28-wide
input to 32, moves the stacked batch across the shim, and later strips the two
useful output columns; all matrix multiplies run on the NPU.
Figure~\ref{fig:model-hardware-mapping} makes that deployment contract
concrete.

\begin{figure}[t]
\centering
\includegraphics[width=\textwidth]{higgs_xdna2_model_hardware_mapping.pdf}
\caption{Model-to-hardware mapping for the retained HIGGS/XDNA~2 deployment
path. The NPU runs the full inference pipeline: one tile for the

The important design choice is not just ``use an MLP'', but ``use an MLP whose
depth and width map directly to the array's tile structure''. That keeps the design
simple enough to analyze and direct enough to deploy.

\section{From Python model to NPU artifact}

The toolchain is deliberately short.
\path{aie_kernels/matmul_relu_skip.cc} defines the tile-local residual kernel,
while \path{aie_kernels/matmul_bias.cc} is compiled into two archives for the
embed and head linear stages. \path{resmlp/streaming_design.py} turns a chosen
$(H, B, L)$ shape into a spatial-streaming MLIR-AIE program that wires embed,
residual body, and head into a single snake. \path{resmlp/streaming_op.py}
wraps the compiled XCLBIN and instruction binary as a resident XRT operator.
Finally, \path{resmlp/streaming_infer.py} drives trained checkpoints through
that operator and measures wall-clock throughput.

The novelty claimed here is not a new compiler or runtime. IRON and MLIR-AIE
already provide the compilation substrate \cite{iron,mlir-aie}; what is new is
the model-to-array mapping, the host/NPU partition, and the end-to-end
measurements of that co-designed path on real HIGGS checkpoints.

A key detail of the current forward path is that the residual weights are
embedded at compile time. Each throughput run therefore builds a
checkpoint-specific streaming artifact. That is not the most general
deployment contract. It is, however, the cleanest way to study the geometry,
routing, and wall-clock behavior of a resident spatial-streaming design.

\section{Training protocol on HIGGS}

The data pipeline uses the public \code{jxie/higgs} mirror of the published
train/test splits \cite{jxie-higgs}. The local cache materializer produces a
fixed 10.5M-example train split and a fixed 500k-example test split. For model
selection, training holds out 100k examples from the train split. That leaves
10.4M examples for optimization. All reported runs use the native
28-dimensional feature vector without any padded-width preprocessing.
Hyperparameter search and checkpoint selection use held-out validation
accuracy. The 500k-example test split is used only for final evaluation of
already-selected models.

HIGGS is also a good benchmark because it sits at the intersection of two real
literatures. In high-energy physics it is a canonical dense-event benchmark
\cite{baldi-higgs}, situated within a broader programme of applying machine
learning to particle-physics classification and reconstruction that is
surveyed in \cite{ml4hep-review}. In tabular machine learning it is one of the standard
large-data problems on which tree ensembles such as XGBoost, LightGBM, and
CatBoost remain strong baselines, while modern deep tabular models---including
attention-based and decision-tree-hybrid approaches---are now more
competitive but not universally dominant \cite{xgboost-kdd,lightgbm-nips,catboost-paper,rtdl-revisiting,grinsztajn-tabular,tabnet,node-paper}.
That means the relevant bar is not ``beat every HIGGS leaderboard'', but
``land in a statistically credible part of the baseline landscape while
preserving a deployment shape the NPU can actually exploit''. Good-enough
accuracy on HIGGS is therefore genuinely worth caring about even if the main
contribution of this work remains systems-oriented.

\subsection{CPU throughput baseline}

To put the NPU throughput numbers in context, we measure the same models on a
laptop CPU using both PyTorch and a decomposed NumPy path that mirrors the
NPU's inference structure (separate embed, sequential residual body, head).

\paragraph{Hardware and software.}
The baseline machine is the same AMD Ryzen AI 9 HX 370 (12 cores / 24 threads,
max 5.16~GHz) used for all NPU measurements, running Linux 6.17. PyTorch
2.10.0 uses 12 intra-op threads (the physical core count); NumPy 1.26.4 is
backed by OpenBLAS with the same default thread pool. No explicit thread pinning
or frequency governor tuning was applied; the measurements reflect the default
out-of-the-box configuration.

\paragraph{Methodology.}
Each benchmark processes 5M samples of the same $H=32$, $L=30$ architecture
using random \code{float32} inputs ($D=28$). Ten warmup calls precede the
timed region. The PyTorch path calls \code{model.forward()} under
\code{torch.no\_grad()}; the NumPy path applies the embed, residual chain, and
head as explicit \code{numpy.matmul} calls. We sweep batch sizes from $B=8$
(matching the NPU's native streaming batch) through $B=1024$ to show how CPU
throughput scales with batch-level parallelism. The model, weights layout, and
input distribution match the deployed NPU pipeline exactly.

Two patterns are worth noting. First, the CPU strongly favors large batches:
PyTorch throughput improves by more than $30\times$ going from $B=8$ to
$B=1024$, because larger batches amortize per-call dispatch overhead and
expose enough work for BLAS multi-threading. Second, the NumPy decomposed
path is faster than PyTorch at small-to-mid batches (lower per-call overhead)
but degrades at $B=1024$ where cache pressure from the $L=30$ residual chain
outweighs the parallelism gain, while PyTorch's fused ATen kernels keep
scaling.

Most relevant to the NPU comparison: at the NPU's native streaming batch size
of $B=8$, the CPU reaches only 147k samples/s at best, while the on-array
pipeline sustains 2.44M samples/s---a $\sim\!17\times$ factor in the NPU's
favor. At each side's best operating point, the CPU tops out at 976k
samples/s while the NPU scales to 14.08M samples/s at $B=64$---a
$\sim\!14\times$ factor, on the identical model. Within the same commodity
chip, pushing the entire residual pipeline onto the on-die array therefore
yields more than an order-of-magnitude throughput advantage over the 12 host
cores.

\paragraph{Note.}
Both the CPU baseline and the NPU throughput measurements were obtained on the
same AMD Ryzen AI 9 HX 370 host, making the comparison controlled: the two
results share the same memory subsystem, OS, and software stack, differing only
in whether the residual body runs on the host cores or the on-die NPU array.

\section{Discussion}

The claim is deliberately narrow but defensible.
\begin{itemize}
  \item A residual MLP can be shaped to match the AMD XDNA~2 array very
        directly: one embed + 30 $H \times H$ residual + one head linear, each
        on its own compute tile, with streamed activations, explicit routing,
        and checkpoint-specific resident weights.
  \item That design delivers 14.08M~samples/s wall-clock at $B=64$ on a real
        collider-style dense benchmark using consumer hardware, while reaching
        99.6\% of the isolated kernel-throughput ceiling.
  \item On the same machine and the identical model, the 12-core CPU peaks at
        0.98M~samples/s, so moving the full pipeline onto the on-die array
        yields roughly a $14\times$ throughput advantage without any change in
        compute substrate.
  \item The same design reaches 76.61\% test accuracy and 0.851 ROC AUC on the
        fixed 500k-example test split, enough to place it inside the credible
        HIGGS baseline band rather than leaving accuracy as a loose end.
\end{itemize}

Just as importantly, the paper avoids claims it does not need. It does not
present the current model as a production trigger deployment, the current
HIGGS result as state of the art, or the present measurements as an
energy-efficiency study. The benchmark is public, the hardware is existing
AMD XDNA~2, and the compiler/runtime substrate is pre-existing; none of those
ingredients are introduced here. The novelty is the co-designed operating
regime: a residual MLP whose full forward path --- embed, residual body,
and classifier head --- maps cleanly onto the 32-tile array so that every
matrix multiply in inference runs on the NPU.

\section{Conclusion and next steps}

The main result is simple: a network designed for spatial compute on AMD
XDNA~2 can deliver unusually high throughput on real dense-event
classification while remaining small enough to understand and validate end to
end. The retained model is a fully on-array $H=32$, $L=30$ residual MLP that
sustains 14.08M~samples/s at $B=64$, stays within 1\% of its own kernel
ceiling, and reaches a credible 76.61\% test accuracy.

The next steps are correspondingly focused. The most direct modeling lever is
to push the on-array model harder on accuracy: longer training schedules,
stronger regularization, and mild architectural tweaks that preserve the
32-tile footprint. On the systems side, genuine kernel-side gains remain
valuable: increasing stream depth toward the per-DMA descriptor limit,
overlapping the device-to-host transfer of one chunk with the host-to-device
transfer of the next, or revisiting the MMUL tiling so that more compute is
packed into each kernel invocation. Energy-efficiency characterization also
remains future work. Those are refinements of an already measured pipeline
rather than structural changes.

\section{The path to 14M samples/s}
\label{app:path-to-14M}

The 14.08M samples/s result in Section~5.2 is the output of a short sequence
of host-side and graph-side changes, not of a single compilation. We record
them here so that a reader interested in reproducing or extending the
measurement can see which ingredients mattered.

\paragraph{Baseline gap.} An earlier revision of this pipeline used a strictly
serial host loop and ran the embed and classifier head on the CPU. Wall-clock
throughput was several times lower than the isolated kernel throughput of the
same compiled bitstream. Instrumenting the host loop showed that the gap was
dominated by Python per-call overhead. For an earlier hybrid $H=64$, $L=32$
configuration, one kernel invocation spent roughly 380~\textmu{}s on the CPU
input projection, 64~\textmu{}s tiling activations, 78~\textmu{}s detiling the
output, and 71~\textmu{}s on the CPU head, against only 156~\textmu{}s in the
kernel wait itself. The host was the bottleneck, not the array.

\paragraph{Collapse the host loop into stacked numpy.} The first change replaced
32 small per-layer Python dispatches per kernel invocation with four stacked
numpy operations: one matmul on all $S \!\times\! B$ rows for the embed, one
reshape/transpose pack, one reshape/transpose detile, and (where still needed)
one stacked head matmul. On the same $H=64$, $L=32$ checkpoint this collapsed
the measured host work from $\sim\!590$~\textmu{}s/call to roughly
116~\textmu{}s/call without touching either the NPU kernel or the network.

\paragraph{Double-buffered host/NPU overlap.} The second change exploited the
fact that the host work and kernel wait are independent: a compiled bitstream
does not care whether the host is already preparing the next chunk. We allocate
two device buffer slots per kernel and run a small producer/consumer loop. At
steady state the host prepares chunk $N{+}1$ (embed, pack, write\_buffer, H2D
sync) into one slot while the NPU is still running chunk $N$ on the other slot;
the post-wait side of the loop then consumes chunk $N$'s output, again in
parallel with the next kernel. Wall time per call becomes close to
$\max(\text{host}, \text{kernel})$ rather than their sum.

\paragraph{Calibrate the kernel separately.} The overlapped loop makes the
per-iteration \texttt{dispatch+wait} interval no longer a clean kernel-time
measurement: it reports whatever the slower side of the pipeline happens to
be. We therefore measure kernel throughput with a dedicated isolated
dispatch+wait calibration on the same compiled bitstream. The ratio between
async wall-clock throughput and that isolated kernel throughput then
quantifies how much of the pure-kernel ceiling the host loop gives back.

\paragraph{Move the entire pipeline on-array.} With host overhead already
amortized, the remaining wall-time cost was the two CPU matmuls that still
bracketed the kernel: the $28 \rightarrow H$ embed and the $H \rightarrow 2$
head. Moving them on-array required shrinking the residual body from 32 to 30
layers so that one tile could host each of the new linear stages. The embed and
head are implemented by a single shared \code{matmul\_bias.cc} kernel compiled
twice with different $(K, N)$ defines, producing two per-stage archives, and
wired into the same snake as the residual body. Because the array now runs the
entire forward pass, host work collapses to zero-padding the 28-wide input to
32 and stripping the two valid columns out of the 8-wide head output.

\paragraph{Scale the streaming batch.} With the full pipeline on-array, the
remaining lever is arithmetic intensity. Going from $B=8$ to $B=64$ packs
$8\times$ more work into each kernel invocation at identical host overhead and
still fits in the $\sim\!64$~KB tile SRAM budget at $H=32$. This is the change
that turns the 2.44M samples/s figure into 14.08M samples/s, because at $B=8$
the retained pipeline is already within 1\% of kernel-limited and the only way
to go faster is to move the kernel-time ceiling itself.

None of these steps change the learned model: the same 76.61\%-accuracy
checkpoint is used throughout. They change the artifact that is actually
dispatched and the way it is dispatched.

\section{HIGGS physics background}

The HIGGS benchmark comes from a standard collider-analysis setting: a rare
target process must be separated from much more common look-alike processes
using reconstructed event kinematics \cite{baldi-higgs}. The label is binary.
In the UCI release, `1' denotes signal and `0' denotes background
\cite{uci-higgs}. Physically, that does not mean the detector has directly
``seen a Higgs.'' It means the reconstructed pattern of visible objects in the
event is more consistent with the target signal process than with ordinary
background production.

Each row is therefore already one reconstructed collision event, not raw
detector data. The low-level variables describe the visible ingredients of the
event: one charged lepton, missing transverse momentum, and up to four jets,
plus per-jet b-tag indicators \cite{uci-higgs}. In collider language, $p_T$ is
momentum transverse to the beam, $\eta$ and $\phi$ encode direction, and
missing transverse momentum records the momentum imbalance associated with
invisible particles leaving the detector. The high-level variables are
invariant-mass combinations such as $m_{jj}$, $m_{jjj}$, $m_{lv}$, $m_{jlv}$,
$m_{bb}$, $m_{wbb}$, and $m_{wwbb}$; they test whether groups of reconstructed
objects line up with a plausible underlying resonance or decay chain
\cite{uci-higgs}.

That structure is exactly why HIGGS is useful here. The model does not have to
solve raw reconstruction. It receives a compact physics summary of each event
and must make a fast signal-versus-background decision on dense numeric inputs.
In this repository those examples come from the public \code{jxie/higgs} mirror
of the published train/test split \cite{jxie-higgs}.
